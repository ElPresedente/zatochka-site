# Тасклист: блок доставки для zatochkaorel.ru

> Контекст: на сайте уже есть магазин, корзина, оформление заказа и оплата, а также
> разделение способа получения на **самовывоз** и **доставку**. Карты на сайте —
> **Яндекс**. Нужно достроить: точку мастерской для самовывоза, зону доставки по Орлу
> (полигон + выбор адреса + расчёт стоимости) и доставку по России через СДЭК
> (виджет ПВЗ + расчёт стоимости, опционально — автосоздание заказа).
>
> **Ключевой принцип UX (как в eShopLogistic):** карта и любые обращения к
> картографическим/логистическим API инициализируются **не на загрузке страницы**, а
> только после того, как пользователь выбрал способ получения и (для доставки) указал
> город/направление. Это и улучшает UX, и резко сокращает число запросов к API, помогая
> оставаться в бесплатных лимитах и не нарушать условия сервисов (см. Этап 2 и Этап 8).
>
> Адрес мастерской (из настроек): **г. Орёл, ул. Полесская д. 2**, координаты
> **lat 52.971119, lon 36.059345**.

---

## Принятые допущения (проверить и при необходимости скорректировать)

- A1. «Сумма заказа» для порога бесплатной доставки по Орлу (≥ 3000 ₽ — бесплатно,
  иначе 200 ₽) считается по **стоимости товаров в корзине** (без услуг заточки).
- A2. По России доставка **всегда платная** по тарифу СДЭК (порога бесплатной нет),
  если клиент не укажет иное.
- A3. Если адрес доставки по Орлу вне полигона — оформление по этому способу
  **блокируется** (с понятным сообщением), а не просто предупреждение.
- A4. Карты везде — **Яндекс.Карты JS API** (единый провайдер с текущим сайтом).
- A5. Стоимость доставки рассчитывается и проверяется **на сервере**; фронт не
  является источником истины по цене.
- A6. **Ленивая загрузка.** До выбора способа получения и направления никакие
  скрипты карт, JS API Яндекса и виджет СДЭК не подключаются и запросы к API
  (suggest, геокодер, тариф, список ПВЗ) не отправляются. Карта/виджет монтируются
  по явному переходу на соответствующий шаг.
- A7. **Минимизация запросов.** Результаты геокодирования и списки ПВЗ кэшируются на
  разрешённый срок (геокодер — до 30 дней по условиям Яндекса); ввод адреса в
  подсказках дебаунсится; при возврате на предыдущий шаг карта/виджет не
  пересоздаются и повторные запросы не шлются.

## Этап 2. Пошаговый UX-флоу и ленивая загрузка карт

> Цель: повторить логику eShopLogistic — сначала выбор, потом карта. Это снижает число
> обращений к API (важно для бесплатных лимитов Яндекса и для соблюдения условий
> сервисов — см. Этап 8).

- [ ] 2.1. **Шаг 1 — способ получения.** Выбор «самовывоз / доставка» обычными
  контролами (radio/кнопки), **без карты и без обращений к API**.
- [ ] 2.2. **Шаг 2 — направление (только для доставки).** Дать выбрать город/направление:
  «по Орлу» или «по России». Это можно сделать полем города или двумя кнопками.
  Значение направления задаёт `delivery_scope` и определяет дальнейший сценарий
  (Орёл → полигон, Этап 4; иначе → СДЭК, Этап 5). Шаг тоже **без карты**.
- [ ] 2.3. **Шаг 3 — карта/виджет.** Карта (Орёл) или виджет СДЭК монтируются **только
  после выбора на шаге 2** и желательно по явному действию пользователя (раскрытие блока
  или кнопка «Выбрать на карте / выбрать ПВЗ»), а не на загрузке страницы.
- [ ] 2.4. **Ленивое подключение скриптов.** Скрипт JS API Яндекса и скрипт виджета СДЭК
  загружать динамически в момент перехода к шагу с картой, один раз за сессию (флаг
  «загружено»), без дублирования. Не держать их в общем `<head>`.
- [ ] 2.5. **Без пересоздания при возврате.** При переключении способа получения /
  направления назад-вперёд — не пересоздавать карту/виджет и не слать повторные запросы;
  переиспользовать уже созданный инстанс (скрывать/показывать контейнер, а не
  инициализировать заново).
- [ ] 2.6. **Состояния загрузки и fallback.** Показывать индикатор загрузки карты; при
  недоступности API (см. Этап 8.5) аккуратно деградировать — давать ручной ввод адреса
  и не блокировать оформление наглухо.
- [ ] 2.7. Обнулять/скрывать неактуальные блоки при смене выбора (например, при переходе
  с «по России» на «самовывоз» скрыть виджет ПВЗ и сбросить связанные поля заказа).

## Этап 4. Доставка по Орлу — полигон + адрес + стоимость

> Карта и подсказки адреса инициализируются только после выбора «доставка → по Орлу»
> на Этапе 2 (A6).

- [ ] 4.1. При «доставка → по Орлу» показать карту Яндекса с **полигоном** зоны доставки
  (координаты полигона — из настроек, см. 7.2).
- [ ] 4.2. Дать выбрать адрес: поле с **подсказками адресов** (Яндекс Геосаджест) и/или
  клик по карте. Выбранный адрес геокодировать в координаты.
  - **дебаунс** ввода (≈300–400 мс) и **минимальная длина** запроса (от 3–4 символов),
    чтобы не слать запрос на каждый символ;
  - **ограничить область** подсказок Орлом и окрестностями (bounded suggest) — меньше
    запросов и точнее результат;
  - **геокодировать только при выборе конкретной подсказки/клике**, а не на каждый ввод.
- [ ] 4.3. **Проверка попадания в полигон** (point-in-polygon) выполнять **на клиенте по
  уже загруженной геометрии полигона** (без дополнительных запросов к API) и продублировать
  на сервере (A5, Этап 8). Если адрес вне зоны — блокировать оформление по Орлу с
  сообщением и предложить самовывоз или доставку по России (см. A3).
- [ ] 4.4. Расчёт стоимости (на сервере, см. A1, A5):
  - сумма товаров **< 3000 ₽** → доставка **200 ₽**;
  - сумма товаров **≥ 3000 ₽** → доставка **0 ₽** (бесплатно).
  Показать стоимость пользователю и добавить в итог заказа.
- [ ] 4.5. Сохранять в заказе адрес и координаты.
- [ ] 4.6. **Кэшировать результат геокодирования** адреса (разрешено до 30 дней,
  Этап 8.3/8.4), чтобы повторный ввод того же адреса не порождал новый запрос.

## Этап 5. Доставка по России — СДЭК

> Используем официальный виджет СДЭК `@cdek-it/widget@3` + серверный прокси `service.php`
> (учётные данные API хранятся **только на сервере**). Виджет рисует карту ПВЗ/постаматов
> на Яндекс.Картах и считает тариф по габаритам/весу корзины; в `onChoose` возвращает
> выбранный ПВЗ и тариф (стоимость + срок). Виджет инициализируется **только** при
> переходе на шаг «по России» (A6).

- [ ] 5.1. Подключить скрипт виджета (`https://cdn.jsdelivr.net/npm/@cdek-it/widget@3`)
  и инициализировать `new CDEKWidget({...})` в контейнере с заданной высотой —
  **лениво, в момент перехода на шаг «по России»** (Этап 2.3–2.4).
  Параметры: `root`, `apiKey` (ключ Яндекс.Карт JS API), `servicePath` (URL прокси),
  `defaultLocation`, `from` = Орёл, `goods` (вес/габариты позиций корзины),
  `onChoose(type, tariff, point)`, `onReady`.
- [ ] 5.2. Развернуть серверный прокси:
  - если backend на **PHP** — взять готовый `service.php` из репозитория `cdek-it/widget`
    (`/dist/service.php`), положить по URL из `servicePath`, прописать учётные данные
    СДЭК (account/secure password из договора) и город-отправитель = Орёл;
  - если backend **не на PHP** — реализовать аналогичный эндпоинт на стеке проекта
    (получение токена СДЭК, проксирование расчёта тарифа и списка ПВЗ);
  - требования к PHP-варианту: PHP ≥ 7.4, расширение cURL.
  - **На время разработки** использовать тестовые учётные данные СДЭК из документации
    виджета, затем заменить на боевые из договора (хранить в env/настройках, не в коде).
- [ ] 5.3. Передавать в виджет реальные **вес и габариты** корзины (из полей товаров;
  при отсутствии — дефолтная упаковка из настроек, пометить расчёт как ориентировочный).
- [ ] 5.4. В `onChoose` сохранять выбранный ПВЗ (код, адрес, город) и стоимость тарифа;
  добавить стоимость в итог заказа. **Перепроверить стоимость на сервере** через API
  расчёта СДЭК перед финальным сохранением заказа (защита от подмены на фронте) —
  **один раз** на финальном шаге, а не на каждое изменение (Этап 8.6).
- [ ] 5.5. Запретить оформление по России без выбранного ПВЗ.
- [ ] 5.6. **Автосоздание заказа в СДЭК** через `POST /v2/orders` после оплаты.
  Договор подтверждён как **«Интернет-магазин»** (в ЛК доступен тип заказа
  «интернет-магазин»), поэтому автосоздание доступно в полном объёме — с магазинными
  тарифами и наложенным платежом.
  - **Явно** указывать `type` = «интернет-магазин» и магазинный тариф. Если по ошибке
    создавать заказы с типом «доставка», получим обычные тарифы без скидок ИМ и без
    наложенного платежа, несмотря на договор ИМ.
  - сохранять `uuid`/номер заказа СДЭК, обрабатывать ошибки, желательно подписаться на
    вебхуки статусов;
  - **Альтернатива/фолбэк:** если на старте автосоздание не включаем — менеджер создаёт
    накладную вручную в ЛК СДЭК, а сайт лишь сохраняет выбранный ПВЗ и стоимость
    (виджет работает независимо от автосоздания).
  - **Наложенный платёж (если нужен):** комиссия СДЭК за перевод НП — 3%. Перед
    использованием подтвердить у менеджера, что НП активирован для договора (сам режим ИМ
    может быть, а приём НП иногда включают отдельно по заявлению).

## Этап 8. Соответствие правилам сервисов и минимизация запросов

> Цель — максимально снизить риск блокировки и нарушения условий бесплатного
> использования. Лимиты и условия меняются (Яндекс пересматривал их в 2025), поэтому
> **актуальные значения сверить в кабинете разработчика** перед запуском.

- [ ] 8.1. **Ленивая загрузка — главный рычаг.** Убедиться, что карты/виджеты и запросы
  к API не выполняются до выбора способа получения и направления (Этап 2, A6). Это
  основной способ удержаться в бесплатных лимитах.
- [ ] 8.2. **Подключить нужные продукты Яндекса к ключу.** После раздельной тарификации
  продукты подключаются по отдельности: **JS API + Геокодер + Геосаджест**. На новом
  ключе по умолчанию активны не все сервисы — без подключения Геокодера/Геосаджеста
  подсказки и геокодирование работать не будут.
- [ ] 8.3. **Соблюдать условия бесплатного использования Яндекса:**
  - сайт должен быть общедоступным (не закрытая система);
  - не скрывать и не видоизменять копирайт и логотип; оставить кнопку «Открыть в Картах»;
  - не использовать API для мониторинга/диспетчеризации транспорта и в игровых проектах;
  - данные нельзя хранить/изменять, но **можно кэшировать** запросы к геокодеру и
    маршрутизатору на срок **до 30 дней**.
- [ ] 8.4. **Кэширование (в пределах разрешённого):**
  - геокодирование адресов Орла — до 30 дней (Этап 4.6);
  - список ПВЗ по городу и тарифы по связке «город + вес + габариты» — на короткий срок,
    чтобы повторные расчёты не порождали новые запросы.
- [ ] 8.5. **Обработка превышения и ошибок API:**
  - при ответах **403/429** от Яндекса (превышение суточного лимита — блок до конца
    суток, не штраф) или ошибке СДЭК: не зацикливать ретраи, показать понятное сообщение
    и fallback на ручной ввод адреса; **не ронять оформление заказа**;
  - логировать такие случаи, чтобы видеть, упираемся ли в лимиты.
- [ ] 8.6. **Не плодить лишние запросы:** дебаунс ввода адреса; пересчёт тарифа/стоимости
  только при изменении значимых параметров (адрес, состав корзины, вес/габариты);
  серверная перепроверка тарифа СДЭК — один раз на финальном шаге (5.4), а не на каждое
  движение.
- [ ] 8.7. **Учитывать лимит частоты (RPS).** Ориентир бесплатной версии Яндекса — до
